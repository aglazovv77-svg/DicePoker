# Обзор кода — DicePoker

---

## SOLID

**Single Responsibility (наиболее критичная проблема)**

Класс `Game` делает всё: бросает кости, определяет комбинации, ведёт счёт, работает с консолью (ввод и вывод) и управляет игровым циклом. Это разные обязанности. Типичное разбиение:
- `Dice` / `DiceRoller` — бросок костей
- `HandEvaluator` — определение комбинации и подсчёт очков
- `ConsoleUI` / `Printer` — все вызовы `System.out`
- `Game` — только оркестрация

**Напрямую связано**: `detectCombination` (строка 155) одновременно и оценивает руку, и выводит название комбинации в `stdout`. Метод, возвращающий значение, не должен иметь побочных эффектов. Вывод должен быть в вызывающем коде.

**Dependency Inversion**

`Game` жёстко привязан к `new Scanner(System.in)` и `new Random()` как к статическим полям. Это делает юнит-тестирование невозможным — нельзя замокать `Random` или поток ввода. Оба должны быть параметрами конструктора (после перевода класса в нестатический).

## DRY

**1. `playerTurn()` и `botTurn()` идентичны**, кроме того, в какой статический массив они пишут (строки 83–134):

```java
private static int[] playerTurn() {
    for (int i = 0; i < 5; i++) PLAYER_ROLL[i] = random.nextInt(6) + 1;
    return PLAYER_ROLL;
}
private static int[] botTurn() {
    for (int i = 0; i < 5; i++) BOT_ROLL[i] = random.nextInt(6) + 1;
    return BOT_ROLL;
}
```

Должен быть один метод: `private static int[] rollDice()`.

**2. `isSmallStraight` и `isLargeStraight` идентичны**, кроме целевого массива (строки 211–221). Один метод `isStraight(int[] dice, int[] expected)` покроет оба случая.

**3. Циклы подсчёта очков для двух пар и одной пары почти одинаковы** (строки 182–196). Единственное отличие — `+=` против `=`. Выделить в `sumPairValues(int[] frequencies)`.

**4. Строка-разделитель** `"___________________________________"` захардкожена в двух ветках `printScore` (строки 230, 233). Использовать константу.

## KISS / Корректность

**1. Сравнение по ссылке как управляющая логика (серьёзный риск бага)**

`printRoll` (строка 91) определяет чей бросок через `roll == PLAYER_ROLL` — сравнение указателей. Это случайно работает сейчас, потому что `rerollSelectedDice` изменяет массив на месте и возвращает ту же ссылку. Если метод когда-либо изменится и начнёт возвращать копию — вывод молча сломается. Нужно передавать метку явно:

```java
private static void printRoll(String owner, int[] roll)
```

**2. Та же проблема в `printScore`** (строка 228): `score == playerScore` использует равенство значений для выбора метки. Если оба счёта окажутся равны — выполнится только ветка игрока, что даст неверный результат для бота. Нужно передавать метку или флаг.

**3. `printDetectCombination` — бессмысленная обёртка с вводящим в заблуждение названием** (строки 223–225). Внутри только `System.out.println(points)`, тогда как название комбинации уже выводится внутри `detectCombination`. Название метода подразумевает вывод комбинации, а не просто числа. Либо объединить с вызывающим кодом, либо переименовать в `printPoints`.

**4. `getRerollChoice` не валидирует ввод Y/N** (строки 103–119). Любой ввод, отличный от `"N"`, попадает в ветку ввода индексов. Если пользователь введёт "может быть", произойдёт `NumberFormatException`. Также нет проверки границ индексов — "0" или "6" вызовет `ArrayIndexOutOfBoundsException`.

**5. Проверка победителя после хода бота** (строки 75–79) использует `if (isPlayer()) ... else if (isBot())`. Если оба достигнут 100 в одном раунде — выведется только сообщение об игроке, а победа бота молча проигнорируется. Эти два условия не являются взаимоисключающими.

**6. `rerollSelectedDice` изменяет аргумент на месте и при этом его же возвращает** (строки 122–127). Вызывающий код может ошибочно считать, что получает новый массив. Нужно выбрать одно: изменять на месте и возвращать `void`, либо возвращать новый массив без изменения оригинала.

## Именование

| Текущее | Проблема | Предложение |

| `PLAYER_ROLL`, `BOT_ROLL` | Стиль ALL_CAPS — для констант; эти поля изменяемые | `playerRoll`, `botRoll` |
| `isPlayer()`, `isBot()` | Читается как "является ли игроком?", а не "выиграл ли игрок?" | `hasPlayerWon()`, `hasBotWon()` |
| `detectCombination` | Делает два дела (определяет + выводит) | `evaluateHand` после удаления побочных эффектов |
| `printDetectCombination` | Вводит в заблуждение — выводит только очки | `printPoints` |
| `diceRoll` (локальная переменная в `start`) | Переменная для результата после переброса названа так же, как сам бросок | `finalRoll` или `rerolledDice` |
| `choiceDiceIndexes` | Многословно | `rerollIndices` |

## Статическое состояние

Всё игровое состояние (`playerScore`, `botScore`, `PLAYER_ROLL`, `BOT_ROLL`, `scanner`, `random`) — статическое. Последствия:
- Игру нельзя сбросить без перезапуска JVM (или ручного обнуления каждого поля).
- Невозможно запустить два экземпляра игры одновременно.
- Невозможно тестировать в изоляции.

Перевести `Game` в обычный (нестатический) класс. Состояние становится полями экземпляра, `Scanner` и `Random` — аргументами конструктора.

## Мелочи / Стиль

- `import java.util.*` — использовать явные импорты (`ArrayList`, `Arrays`, `Collections`, `List`, `Random`, `Scanner`).
- Javadoc отсутствует на всех публичных методах. Как минимум `start()`, `printGreeting()`, `inputCommand()`, `isQuit()` должны иметь краткий Javadoc.
- Магические числа в `detectCombination` (15, 25, 50, 35, 30, 20) — выделить в именованные константы, например `SMALL_STRAIGHT_POINTS = 15`.
- `playerTurn()` и `botTurn()` оба модифицируют статический массив *и* возвращают его — возвращаемое значение избыточно, так как вызывающий код и так имеет доступ к статическому полю. Нужно выбрать один способ передачи данных.
